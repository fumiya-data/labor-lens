このリポジトリは、実務向けの業務アプリケーションを想定して設計している
ポートフォリオ作品です。

架空またはサンプルの業務データを前提に、要求定義、業務ルール、設計、
検証計画、実装計画を順に残しながら、コード生成より前の工程も含めて
開発プロセスを示すことを目的としています。

# 製品の方向性

LaborLens は、勤怠、疲労、人件費、売上データを組み合わせる労務分析
アプリケーションです。健康、法令遵守、プライバシー、公平性を強い制約として扱います。

本番構成は、外部公開する Web サービスではなく、利用者のPCまたは社内端末で動く
ローカルサーバー、ローカルDB、バックグラウンドジョブ、ローカルUIを前提にします。

本番実装は、試作品コードをそのまま移植するのではなく、Rust で再設計します。
試作品は、要件、スキーマ、検証ケース、レポート形状、配布上の教訓を得るための
証拠として扱います。

# 初期本番スタック

| レイヤー | 方針 |
| --- | --- |
| Local UI | ローカルサーバーに接続し、実行指示、進捗確認、結果閲覧を行う |
| Local server | CSV 取り込み、ジョブ管理、レポート配信、ローカルDB接続を担当 |
| Local DB | 原本 metadata、正規化データ、issue、集計結果、実行履歴を保存 |
| Background jobs | 重い CSV 検査、集計、レポート生成を画面操作から分離して実行 |
| Core logic | Rust |
| CSV ingest and validation | Rust |
| Metrics and calculation engine | Rust |
| Report model and machine-readable output | Rust |
| Optional rich report rendering | 必要な場合のみ薄い Python レイヤー |
| Local assistant / RAG | コアワークフロー安定後に Ollama ベースのローカル実行環境 |

# 製品制約

- 原本 CSV ファイルを変更してはいけない。
- 顧客の日本語 CSV ヘッダーは、安定した内部フィールド名へマッピングする。
- 売上データは日付と時間帯の粒度を扱える必要がある。
- 人件費データは、従業員別月次、部門別月次、雇用区分別月次など複数の粒度を扱える必要がある。
- 個人の疲労値を平文レポートやユーザー向け画面に表示してはいけない。
- 生成されるすべての成果物には、安定した run identifier を含める。
- データ品質の指摘と業務上の推奨は分離する。
- 10000人規模・3年分の勤怠データを扱える設計を目標にする。
- 重い処理はローカルサーバー側のバックグラウンドジョブとして実行する。
